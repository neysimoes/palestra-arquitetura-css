Metas da boa arquitetura CSS

Previsível

As regras são claras ao ponto dos comportamentos serem previsíves quando utilizados. Adicionar mais estilos deve ser feito de forma fluida seguindo o padrão de toda a aplicação. Alterar ou adicionar estilos não deve afetar outras partes da aplicação.

Reutilizável

As regras devem ser abstraídas e desassociadas ao ponto de você poder reutiliza-las. Usando estilos já existentes para novos componentes.

Sustentável

Quando precisar alterar, adicionar ou reorganizar o conteúdo do seu site, isso não deve exigir refatoração no código existente. Adicionar um novo componente a uma página existente não deve afetar os componentes existentes.

Escalável

É difícil admitir, mas o projeto cresce e mais desenvolvedores serão adicionados a sua equipe. Não importa se é uma ou cem pessoas o CSS deve ser facilmente gerido por todos envolvidos.

Erros comuns

Estilizar usando elemento pai

O erro comum é usar o emento pai para determinar o contexto.


.component {
	background: green;
	border: 1px solid #ddd;
	border-radius: 3px;
}
#header .component {
	height: 50px;
}
div.menu .component {
	background: yellow;
}

Complicar sem necessidade


#header .menu > li { ... }

#header .menu li li .link { ... }

Entendendo as partes

Bloco ou componente

Um bloco é uma entidade independente, um "bloco de construção" de uma aplicação. Um bloco pode ser simples ou composto (contendo outros blocos).

Elemento

Um elemento é uma parte de um bloco que executa uma certa função. Elementos são dependentes do contexto: eles só fazem sentido no contexto do bloco a que pertencem.

Modificador

O uso do modificador é opcional. Basicamente você cria um bloco e mudando o modificador você muda a aparência dele.

Soluções

Convenção de no menclatura

Usando a metodologia BEM como base

Algumas regras são necessárias

  • Nomes são escritos em caixa baixa usando letras e números
  • Nomes compostos serão separados por hífen (-)
  • Nunca use IDs ou o elemento para estilização apenas classes.

Blocos

Blocos são únicos por isso seus nomes também devem ser. Pode ser adicionado um prefixo para facilitar na hora da nomenclatura. A classe é o próprio nome.


.nome-do-bloco {...}

/* Exemplos */
.menu { ... }
.widget-facebook { ... }

Elementos

Seu nome deve ser único dentro do contexto do bloco, mas pode se repetir de de um bloco para outro. A classe é o nome do bloco seguido pelo nome do elemento separado por dois sublinhados (__).


.nome-do-bloco__nome-do-elemento {...}

/* Exemplos */
.menu__item { ... }
.menu__link { ... }
.menu__sub-menu { ... }
.menu__sub-item { ... }

Modificadores

O nome do modificador é composto pelo nome do bloco ou elemento há que pertence. O nome do modificador é delimitado com um único sublinhado (_) ou dois hífens (--).


.bloco_modificador {}
.bloco--modificador {}


/* Errado */
#footer .menu { ... }

/* Correto */
.menu--footer { ... }
.menu__item--first { ... }

Organização de arquivos

Usando a metodologia SMACSS como base

Cinco principais categorias:

  • Base
  • Layout (.l)
  • Module
  • State (.is-)
  • Theme (placeholder)

Siga uma estrutura lógica de nomenclatura


/* Template */
%template-name { }
%template-name--modifier-name { }
%template-name__sub-object { }
%template-name__sub-object--modifier-name { }

/* Component */
.component-name { }
.component-name--modifier-name { }
.component-name__sub-object { }
.component-name__sub-object--modifier-name { }

/* Layout */
.l-layout-method { }
.grid { }

/* State */
.is-state-type { }

/* Non-styled JavaScript Hooks */
.js-action-name { }

Conclusão

CSS não é apenas design visual. Não jogue fora as melhores práticas de programação só porque você está escrevendo CSS. Conceitos como OOP, DRY, etc. também se aplicam ao CSS.

A base de tudo é como você faz para organizar o seu código. Seja crítico com seu código e analize se é algo vai realmente te ajudar a tornar seu desenvolvimento mais fácil e sustentável a longo prazo.

Não existe bala de prata, analize cada caso e veja o que é melhor para cada projeto.
